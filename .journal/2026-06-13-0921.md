## Slim Core, Opt-in Streaming, and Explicit Remote References

### Timestamp

2026-06-13 09:21

### Core Decision/Topic

Reworked the stable kkrpc architecture so the default `kkrpc` / `kkrpc/browser` core returns to the small pre-streaming bundle profile while keeping the newer high-level capabilities available through opt-in subpath entries.

The main outcome is a three-tier feature split:

1. **Default core (`kkrpc`)** â€” small request/response RPC channel with proxy API, plugins, transfer descriptors, property access, constructors, and legacy top-level fire-and-forget callback arguments.
2. **Streaming entry (`kkrpc/streaming`)** â€” async iterable arguments/results with pull-based backpressure and stream cleanup semantics.
3. **Remote reference entry (`kkrpc/remote-refs`)** â€” Comlink-style explicit `proxy(value)` references, nested explicit markers, object proxies, callback return values, and deterministic `releaseProxy()` cleanup.

This changes the feature packaging, not the general transport architecture. Existing native transports still expose `Transport<RPCMessage>` and can be used with the appropriate channel entry.

### Problem Context

Recent features had been added directly to `RPCChannel`, causing the default browser bundle to grow substantially:

- Before async iterable streaming, `kkrpc core` measured around **6.17 KB raw / 2.36 KB gzip**.
- Adding async iterable streaming moved core to roughly **12.85 KB raw / 3.99 KB gzip**.
- Adding remote references directly to default `RPCChannel` moved core to roughly **22.01 KB raw / 6.09 KB gzip / 5.44 KB brotli**.

That size profile was not aligned with the desired default package shape. The project wants a small core for ordinary RPC and explicit opt-in cost for advanced behavior.

### Options Considered

#### Option 1 â€” Keep all features in default `RPCChannel`

This would preserve the most feature-rich default experience. Users could pass async iterables, returned handles, and callback functions without thinking about entry selection.

Rejected because it makes every consumer pay for streaming and remote-reference state machines, even when using simple request/response RPC. It also makes browser bundle comparisons misleading because optional features are hidden in the core row.

#### Option 2 â€” Create separate plugins layered onto one channel

This could keep one public `RPCChannel` while making streaming and remote references configurable.

Deferred because these features affect core encode/decode, request routing, pending request handling, lifecycle cleanup, and protocol message dispatch. A plugin-only approach would either require many fragile hooks or keep most code in core anyway.

#### Option 3 â€” Split advanced features into opt-in channel subclasses and entries

Chosen approach. Keep default `RPCChannel` small and use subclasses for featureful entries:

- `StreamingRPCChannel` in `kkrpc/streaming`
- `RemoteReferenceRPCChannel` in `kkrpc/remote-refs`

This makes feature costs explicit, keeps common imports small, and gives documentation a clean story: choose the entry that matches the behavior you need.

### Final Decision & Rationale

The default `kkrpc` entry now intentionally does **not** include async iterable streaming or request/response remote references. Those features are not removed; they are moved to dedicated subpaths.

The rationale:

- **Bundle size:** The default browser-safe core is back near the original ~6 KB raw baseline.
- **API clarity:** Advanced behavior is no longer surprising. If code needs callback return values or long-lived handles, it imports `kkrpc/remote-refs`. If it needs `AsyncIterable`, it imports `kkrpc/streaming`.
- **Comlink-style explicitness:** Remote references now require `proxy(value)` for the slim explicit entry. Plain data remains by-value. Nested explicit markers inside plain arrays/objects are supported, but unmarked nested functions are not auto-proxied.
- **Future flexibility:** Automatic nested function refs can still be added later as a separate opt-in feature or entry without bloating core.

### Important Semantics After the Split

#### Default core (`kkrpc`)

- Supports normal request/response calls.
- Supports property get/set and constructors through the proxy API.
- Supports top-level callback arguments using compact `t: "cb"` callback messages.
- Callback arguments in default core are **fire-and-forget**; callback return values and callback-thrown errors are not propagated to the remote caller.
- Does not encode async iterables as streams.
- Does not encode `proxy()` remote-reference envelopes.

#### `kkrpc/streaming`

- Supports async iterable return values and async iterable arguments.
- Uses `t: "sq"` stream control/credit records and `t: "sr"` stream result records.
- Uses bounded pull credit instead of one round trip per chunk.
- Closes decoded remote streams when a remote call fails before consuming them.

#### `kkrpc/remote-refs`

- Requires explicit `proxy(value)` to pass functions or objects by reference.
- Supports explicit markers nested inside plain arrays/objects.
- Does not automatically proxy unmarked nested functions.
- Supports callback return values and thrown callback errors because proxied functions are invoked via request/response semantics.
- Supports `releaseProxy(value)` for deterministic remote handle cleanup.

#### HTTP

- Remains unary request/response only.
- Rejects callback arguments, stream refs, async iterable streams, and remote refs because those require follow-up bidirectional traffic.
- Remains appropriate for ordinary value-only RPC calls.
